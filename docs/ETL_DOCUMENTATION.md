# ETL Pipeline Documentation — HDMS Phase 2

## Overview

The ETL (Extract, Transform, Load) pipeline ingests historical helpdesk ticket data from a CSV file, cleans and standardises it, and loads it into a dedicated MySQL analytics table. It runs independently of the Phase 1 tickets table and does not modify any existing data.

```
tickets_historical.csv
        │
        ▼  EXTRACT
   Raw DataFrame (250 rows)
        │
        ▼  TRANSFORM
   Cleaned DataFrame (230 rows)
   - duplicates removed
   - values standardised
   - computed columns added
        │
        ▼  LOAD
   tickets_analytics  (MySQL)
   etl_meta           (MySQL)
```

## Source File

| Property        | Value                                      |
|-----------------|--------------------------------------------|
| File            | `backend/data/tickets_historical.csv`      |
| Rows            | 250 (including 20 intentional duplicates)  |
| Generated by    | `backend/data/generate_dataset.py`         |
| Date range      | Last 6 months from generation date         |

### CSV Columns

| Column             | Type     | Description                                      |
|--------------------|----------|--------------------------------------------------|
| `employee_name`    | string   | Full name from a pool of 45 Indian names         |
| `department`       | string   | IT / HR / Finance / Operations / Marketing / Engineering / Sales |
| `issue_category`   | string   | One of 7 categories (see below)                  |
| `description`      | string   | Short description matching the category          |
| `priority`         | string   | Weighted: 40% Low, 30% Medium, 20% High, 10% Critical |
| `status`           | string   | Weighted: 20% Open, 20% In Progress, 40% Resolved, 20% Closed |
| `created_at`       | datetime | Random date within last 6 months                 |
| `resolution_notes` | string   | Populated only when status is Resolved or Closed |

## Extract Stage

**Script:** `backend/etl/etl_pipeline.py` → `extract()`

- Reads `tickets_historical.csv` using `pandas.read_csv()`
- Logs total row count: `"Extracted X rows from CSV"`
- Returns a raw `DataFrame` with no transformations applied

---

## Transform Stage

**Script:** `backend/etl/etl_pipeline.py` → `transform()`

### Cleaning Rules Applied

| Rule                     | Detail                                                              |
|--------------------------|---------------------------------------------------------------------|
| **Deduplication**        | `DataFrame.drop_duplicates()` — removes 20 exact duplicate rows    |
| **Category standardise** | Explicit lookup map preserving acronyms (e.g. VPN, not Vpn)        |
| **Priority standardise** | Case-insensitive map → `Low / Medium / High / Critical`            |
| **Status standardise**   | Case-insensitive map → `Open / In Progress / Resolved / Closed`    |
| **Resolution notes**     | `NaN` filled with empty string `""` for MySQL compatibility        |
| **Date parsing**         | `created_at` parsed to `datetime` via `pd.to_datetime()`           |

### Computed Columns Added

| Column                | Logic                                                         |
|-----------------------|---------------------------------------------------------------|
| `resolution_time_days`| Random int 1–14 if status is Resolved/Closed, else `NULL`    |
| `ticket_source`       | Hardcoded `"historical_import"` for traceability             |
| `imported_at`         | `datetime.utcnow()` at time of pipeline execution            |

Logs: `"Transform complete. X rows after cleaning."`

## Load Stage

**Script:** `backend/etl/etl_pipeline.py` → `load()`

### Target Table: `tickets_analytics`

| Column                | Type          | Notes                          |
|-----------------------|---------------|--------------------------------|
| `id`                  | INT PK        | Auto-increment                 |
| `employee_name`       | VARCHAR(100)  |                                |
| `department`          | VARCHAR(100)  |                                |
| `issue_category`      | VARCHAR(100)  |                                |
| `description`         | TEXT          |                                |
| `priority`            | VARCHAR(20)   |                                |
| `status`              | VARCHAR(30)   |                                |
| `resolution_notes`    | TEXT          | Empty string if unresolved     |
| `created_at`          | DATETIME      |                                |
| `resolution_time_days`| INT           | NULL for open/in-progress      |
| `ticket_source`       | VARCHAR(50)   | `"historical_import"`          |
| `imported_at`         | DATETIME      | UTC timestamp of pipeline run  |

The table is **dropped and recreated on every run** (demo mode). This ensures the analytics data always reflects the current CSV.

### Metadata Table: `etl_meta`

Tracks the most recent pipeline run (single row, truncated and replaced each run):

| Column              | Type     | Description                     |
|---------------------|----------|---------------------------------|
| `id`                | INT PK   |                                 |
| `run_at`            | DATETIME | UTC time the pipeline ran       |
| `rows_extracted`    | INT      | Rows read from CSV              |
| `duplicates_removed`| INT      | Rows removed in deduplication   |
| `rows_loaded`       | INT      | Final rows inserted to DB       |

Logs: `"Loaded X rows into tickets_analytics"`

## How to Re-run the Pipeline

### Via CLI (recommended for first run)

```bash
cd backend
python etl/run_etl.py
```

### Via API (from the frontend or Postman)

```http
POST http://localhost:8000/etl/run
```

Response:
```json
{
  "extracted": 250,
  "duplicates_removed": 20,
  "loaded": 230,
  "status": "success"
}
```

### Via the Analytics Dashboard

Click the **Re-run ETL** button in the top-right of the `/analytics` page. The page will reload automatically after the run completes.

## Architecture Notes

- The ETL pipeline creates its own SQLAlchemy engine directly from environment variables — it does not use the FastAPI `get_db()` session dependency.
- The `tickets` table (Phase 1) is **never touched** by the ETL pipeline.
- All analytics API endpoints read exclusively from `tickets_analytics`.
- `pandas` NaN values are explicitly converted to `None` before MySQL insertion to avoid `ProgrammingError: nan can not be used with MySQL`.
